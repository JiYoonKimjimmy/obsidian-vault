
### AWS 활용 능숙도를 보여주실 수 있는 경험들을 더 자세히 설명해 주실 수 있을까요?

현재 사내에서는 온프레미스 환경의 인프라 구성을 표준으로 하고 있으나, "가상 순번 대기 시스템 신규 개발" 프로젝트를 통해 AWS 클라우드 인프라를 처음으로 구축한 경험이 있습니다.

**[프로젝트 배경]**
지역화폐 충전 인센티브 지급 등 일시적으로 폭증하는 트래픽 대응을 위한 가상 순번 대기 시스템 개발이 필요했습니다.
기존 온프레미스 환경으로는 동적 확장이 어려워 대규모 트래픽 처리에 한계가 있었고, AWS 클라우드 인프라를 활용하여 확장 가능하고 안정적인 시스템을 구축해야 했습니다.

**[인프라 구성]**
대용량 트래픽 제어를 위해 AWS의 관리형 서비스들을 활용하여 다음과 같은 인프라를 구성했습니다:

![Mermaid Diagram](/images/experience_spoqa-0.png)

**[주요 구현]**
- **ALB를 통한 로드 밸런싱**: Application Load Balancer를 활용하여 트래픽을 여러 EC2 인스턴스에 분산 처리
- **EC2 Auto-Scaling**: 트래픽 증가에 따라 자동으로 인스턴스를 확장/축소하여 리소스를 효율적으로 관리
- **Redis를 활용한 대기열 관리**: Redis Sorted-Set 기반으로 대기열 및 순번 관리, Sliding-Window로 트래픽 대기/진입 처리 제어
- **AuroraDB를 활용한 모니터링 데이터 저장**: 대기 현황 모니터링 데이터를 AuroraDB에 저장하고, 모니터링 데이터 조회 인스턴스를 통해 백오피스에서 시각화

**[성과]**
- 동시접속자 최소 100K 이상 트래픽 대기 및 제어 가능한 분산 시스템 구축
- AWS 클라우드 인프라를 처음으로 구축하여 클라우드 환경에 대한 실무 경험 확보
- Auto-Scaling 환경 구성으로 트래픽 폭증 시 자동 확장하여 안정적인 서비스 제공

### 사용자의 피드백을 통해 서비스를 개선하거나 사용자의 문제를 해결하기 위해 제품을 개선한 경험이 있다면 구체적으로 설명 부탁드립니다.

핀테크 보안 점검 팀의 개선 제안을 통해 회원 로그인 인증 프로세스를 강화한 경험이 있습니다.

**[개선 배경]**
보안 점검 팀에서 "회원 로그인하기 위한 인증 단계 별 완료 여부 저장 필요"라는 개선 항목을 제안받았습니다.
기존에는 각 인증 단계의 완료 여부를 체계적으로 관리하지 않아, 필수 인증 단계를 모두 완료하지 않은 상태에서도 로그인이 가능한 보안 취약점이 존재했습니다.

**[인증 프로세스]**
회원 로그인을 위해서는 다음 3가지 인증 단계를 모두 완료해야 합니다:
1. **통신사 인증**: 본인 확인을 위한 통신사 인증
2. **계좌 인증(1원 인증)**: 계좌 소유 확인을 위한 1원 입금 인증
3. **PIN 6자리 비밀번호 인증**: 최종 보안을 위한 PIN 비밀번호 인증

![Mermaid Diagram](/images/experience_spoqa-1.png)

**[구현 내용]**
- **인증 상태 캐싱**: 각 인증 단계에서 인증 요청 시 "인증 요청" 상태를 캐싱하고, 인증 완료 시 "인증 완료" 상태를 캐싱하여 실시간으로 인증 진행 상황을 추적
- **Login API 검증 로직**: Login API 처리 시 해당 회원이 필수적으로 인증해야 하는 모든 단계(통신사 인증, 계좌 인증, PIN 인증)를 완료했는지 캐시 데이터를 기반으로 검증
- **보안 강화**: 필수 인증 단계를 모두 완료한 회원만 로그인할 수 있도록 하여 보안 취약점을 해소

**[성과]**
- 핀테크 보안 점검 기준을 충족하여 보안 수준 향상
- 인증 단계별 완료 여부를 체계적으로 관리하여 추적 가능성 확보
- 필수 인증 단계를 우회한 로그인 시도 차단으로 보안성 강화

### 이직을 하시려는 이유, 그리고 스포카에서 어떠한 기여를 하며 성장하고 싶으신지 궁금합니다.

**[이직 사유]**
현재 조직에서 약 5년간 백엔드 개발자로 다양한 프로젝트를 수행하며 성장해왔습니다.
특히 "가상 순번 대기 시스템" 프로젝트를 통해 AWS 클라우드 인프라를 처음으로 구축하는 등 기술적 도전을 통해 성장하는 경험을 했습니다.

하지만 최근 몇 년간은 기술적 성장에 대한 갈증을 느끼게 되었습니다.
새로운 기술을 도입하거나 개발 문화를 개선하려는 제안을 할 때, "기존 방식이 안정적이다", "변경에 따른 리스크가 있다"는 이유로 제한되는 경우가 많았습니다. 코드 리뷰나 기술적 토론을 통해 함께 성장하는 문화보다는, 일정과 기능 구현에 집중하는 문화가 강했습니다.

개발자로서 더 깊이 있는 기술 역량을 쌓고, 기술적 의사결정에 주도적으로 참여하며, 팀원들과 함께 성장할 수 있는 환경을 찾고 싶었습니다.

**[스포카에서의 기여와 성장 계획]**
제가 그동안 쌓아온 백엔드 개발 경험과 시스템 설계 역량을 바탕으로 스포카에 기여하고 싶습니다.

특히 "가상 순번 대기 시스템" 프로젝트처럼, 쉬운 해결책이 아닌 어려운 문제를 해결하며 더 큰 성취감을 느끼는 경험을 해왔습니다. 대규모 트래픽 처리, 분산 시스템 설계, 클라우드 인프라 구축 등의 경험을 살려 스포카의 서비스가 더 안정적이고 확장 가능하도록 기여하고 싶습니다.

또한 새로운 도전과 그에 따른 책임을 지는 것에 두려워하지 않고, 스포카의 빠르게 변화하는 환경에 적응하며, 팀원들과 솔직한 피드백을 주고받으며 함께 성장하고 싶습니다.

단순히 기능을 구현하는 것을 넘어, "왜 이렇게 설계해야 하는가"에 대한 본질적인 고민을 함께하며 지속적으로 성장하는 개발자가 되고 싶습니다.
